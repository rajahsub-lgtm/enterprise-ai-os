\# EAIOS 1 Portability Review



\## Purpose



This document reviews the work completed in EAIOS 1 and classifies which parts are reusable EAIOS Core concepts and which parts are specific to the IT application-health domain.



The goal is to complete the application-health story while preserving a clean boundary between:



\* EAIOS Core

\* IT Application Health Domain Adapter

\* Runtime/demo artifacts

\* Future domain adapters such as Financial AML



This review supports ADR-013: Domain Adapter Boundary.



\---



\# Summary Finding



EAIOS 1 is architecturally portable, but its first implementation is IT-domain-specific.



The reusable pattern is:



```text

Goal

↓

Gather signals

↓

Analyze context

↓

Generate recommendation candidate

↓

Require human approval

↓

Record execution

```



The first domain implementation used IT application-health language:



```text

application

service

incident

alert

health

business impact

recommendation

remediation

human approval

```



These terms should not become permanent EAIOS Core vocabulary.



They should live inside the IT Application Health Domain Adapter.



\---



\# Portability Principle



EAIOS Core should reason over domain-neutral concepts:



\* case

\* goal

\* entity

\* observation

\* evidence

\* hypothesis

\* risk

\* control

\* recommendation candidate

\* approval

\* audit



Domain adapters translate domain-specific language into those core concepts.



For IT application health:



| IT Concept            | Core Concept                     |

| --------------------- | -------------------------------- |

| Application / service | Entity                           |

| Incident / alert      | Observation                      |

| Event cluster         | Observation cluster              |

| Telemetry             | Observation source               |

| Runbook / known error | Evidence source / knowledge item |

| Business impact       | Risk / impact                    |

| Root cause candidate  | Hypothesis                       |

| Remediation proposal  | RecommendationCandidate          |

| Change approval       | Human approval                   |

\# EAIOS 1 Runtime Files



EAIOS 1 created runtime files under:



```text

data/memory/

```



Known files:



```text

data/memory/application\_health\_memory.json

data/memory/execution\_history.json

```



\## Classification



| File                                         | Classification                                    | Reason                                                                                                               |

| -------------------------------------------- | ------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |

| `data/memory/application\_health\_memory.json` | IT Application Health Runtime Artifact            | Contains application-health-specific runtime memory. It should not become cross-domain core memory without redesign. |

| `data/memory/execution\_history.json`         | Runtime Execution Artifact with IT-Domain Content | The execution-history pattern is reusable, but the current records are generated by the IT application-health demo.  |



\## Decision



These files are not EAIOS Core contracts.



They should be treated as generated runtime/demo artifacts.



They should not be committed when regenerated by local demo runs unless intentionally updating a controlled fixture.



Future cross-domain memory should use a domain-neutral structure such as:



```text

data/runtime/

&#x20; core\_audit/

&#x20; evidence/

&#x20; case\_history/



data/runtime/domain/

&#x20; it\_application\_health/

&#x20; financial\_aml/

```



For now, avoid refactoring these files until the application-health demo is stabilized.

\# EAIOS 1 Core-Reusable Concepts



The following EAIOS 1 ideas are reusable as core concepts, even if the original implementation used IT language.



| EAIOS 1 Concept           | Core Classification | Notes                                                              |

| ------------------------- | ------------------- | ------------------------------------------------------------------ |

| Goal-driven execution     | Core                | Every domain starts from a governed goal.                          |

| Agent orchestration       | Core pattern        | Agents may vary by domain, but orchestration is reusable.          |

| Context gathering         | Core pattern        | Domain adapters decide what signals mean.                          |

| Reasoning trace           | Core                | Execution trace should be domain-neutral.                          |

| Recommendation generation | Core contract       | Should become RecommendationCandidate, not domain-specific action. |

| Execution history         | Core pattern        | Current data is IT-specific, but the pattern is reusable.          |

| Memory/logging pattern    | Core pattern        | Needs domain-neutral redesign before reuse.                        |

\# Important Boundary: Core Pattern vs Domain Vocabulary



Some code may be structurally core but currently includes IT-specific vocabulary.



Example IT safety tokens:



```text

restart production

disable monitoring

shutdown service

force deploy

```



These are not universal core concepts.



For AML, equivalent high-risk action tokens might be:



```text

freeze account

block transaction

file SAR

close customer account

report to regulator

```



Therefore, the long-term architecture should support:



```text

ContentSafetyGateway

&#x20; + core safety rule engine

&#x20; + domain-specific safety vocabulary

```



For now, keep Sprint 2 tests stable. Add domain-specific safety extensions later through adapters or policy configuration.

\# Immediate Decision



Complete the IT application-health story first.



Use real external RCA-style data through an IT domain adapter.



Do not jump to AML until the IT story is demo-ready.



The next application-health work should use this boundary:



```text

External RCA dataset

↓

IT Application Health Domain Adapter

↓

EAIOS Core Case / Entity / Observation / Evidence contracts

↓

Governance Broker / AGS

↓

EvidenceFactory / ContentSafetyGateway

↓

KT Case Frame

↓

Hypothesis candidates

↓

Recommendation candidates

↓

Human approval

```

\# What Not To Do



Do not put PetShop-specific logic into EAIOS Core.



Do not put application-health vocabulary into core contracts.



Do not use `data/memory/application\_health\_memory.json` as the pattern for cross-domain memory.



Do not refactor `demo.py` before we have a replacement demo and passing tests.



Do not add AML until the IT application-health story is complete enough to demonstrate.



\---



\# Review Conclusion



EAIOS 1 successfully demonstrated the first application-health reasoning story.



Its core architectural pattern is portable.



Its runtime files, demo script, and domain vocabulary are IT application-health specific.



Going forward, EAIOS should complete the IT story through an explicit IT Application Health Domain Adapter while preserving the core/domain split defined in ADR-013.



The goal is not to abandon the IT story.



The goal is to complete it in a way that proves EAIOS Core can later support other domains.
